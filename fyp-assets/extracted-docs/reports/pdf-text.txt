1.INTRODUCTION
1.1 Global Energy and Environmental Challenges
The rise in global population has triggered a surge in energy demand, especially in
transportation. Currently, transportations rely heavily on fossil fuels, which contribute to
greenhouse gas emissions and global warming.
To reduce emissions, we need to find alternative to internal combustion engine (ICE) fuel.
Renewable energy sources like wind and solar but they can only generate energy and cannot
act as energy storage.

1.2 Limitations of Current Electric Vehicle Technology
Currently electric vehicles (EVs) powered by lithium-ion batteries is the primary alternative
to fossil fuel vehicles. However, there are several limitations that slows down the adoption of
EVs. Traditional charging methods are time-consuming. Lithium-ion batteries also suffer
from degradation and have limited life cycles. There is also concerns about the safety risk of
EVs, including battery overheating and ignition upon impact. These challenges make many
users hesitant to swap their ICE vehicles to EVs, we need an alternative solution that
combines the low-emission benefits of EVs with the rapid refuelling and safe to drive.

1.3 The Vanadium Fuel Cell Solution
To address these limitations, this project explores the application of Vanadium Redox Flow
Battery (VRFB) adapted into a passive direct liquid fuel cell stack. Unlike conventional
batteries, vanadium fuel cells have a liquid electrolyte that allows for rapid refuelling.
Furthermore, vanadium electrolytes are non-flammable, offering safety advantages over
lithium battery. The system also has long cycle life and high recyclability, making it a
sustainable candidate for future mobility solutions.

1.4 Project Overview: The "Vandi" Autonomous Vehicle
The objective of this project is to design, and fabricate a electric toy car, named "Vandi,"
powered by passive vanadium fuel cell stack.
This interim report covers the initial design and fabrication phases:
Power System: Design and prototyped a vanadium fuel cell stack with modular design.
Mechanical Integration: Assembly of a mecanum-wheel chassis designed to carry the fuel
cell stack, control system, and sensor.
Intelligent Control: AI navigation systems, depth cameras utilizing YOLO (You Only Look
Once) algorithms for object detection, alongside a prototype for a Large Language Model
(LLM) interface for user interaction.

2.3 The "E-Fuel" Concept and Passive Fuel Cells
Recent innovations have allowed VRFB technology for portable use. Shi et al. introduced the
concept of "electrically rechargeable liquid fuel" (e-fuel), where the charged vanadium
electrolyte 𝑉 2+ serves as a liquid fuel that can refill instantly.
Using this for small-scale design, researchers developed passive direct liquid fuel cells.
Unlike active systems, passive cells do not need pumps and blowers. Instead, it relies on
diffusion and natural convection.

2.4 Autonomous Navigation in Small-Scale Vehicles
To demonstrate a small-scale smart vehicle, we decide to include intelligent navigation.
Modern autonomous navigation relies on object detection models like YOLO that collaborate
with an algorithm for path planning.
YOLO (You Only Look Once): The YOLO algorithm is a real-time object detection model
with high speed and accuracy. For devices like small toy cars, there is also a lightweight
version such as YOLOv11-Nano. These models can process inputs from cameras with high
inference speeds, suitable for computing devices like the Jetson Nano.
Sensor: Effective navigation requires collecting data from depth cameras and LiDAR. This
combination allows the vehicle to map its environment (SLAM) and execute path planning.

The performance of the vanadium redox flow battery is critically dependent on the chemical

and vanadyl sulfate (𝑉𝑂𝑆𝑂4 ) in the vanadium electrolyte solution.
The preparation and testing process involved generating the active vanadium species
(𝑉 2+ 𝑎𝑛𝑑 𝑉 5+ ) from an initial vanadyl sulfate (𝑉𝑂𝑆𝑂4 ) precursor through a specific charging
protocol.

4.4.5 HSV tuning for traffic light state detection

Figure 14 Testing the HSV value inside a live HSV tuner.

YOLO is great at object detection, but to determining the state of the light, we need to use
traditional computer vision techniques to achieve higher accuracy.
Once YOLO identifies the bounding box of a traffic light, that region of interest (ROI) is
processed using HSV (Hue, Saturation, Value) colour filtering.
By counting the number of pixels falling into each colour mask, the system deterministically
identifies the signal state. This hybrid approach combines the robust localization of deep
learning with the computationally efficient colour discrimination of classical computer
vision.

Figure 19 AI chat interface demo.

To enhance the demonstration and user engagement, an interactive chatbot interface was

22

4.6.1 Interactive chatbot logic flow

Figure 20 AI chat bot logic flow.

The system architecture is designed to decouple the heavy AI processing from the real-time
vehicle control. The logic flow operates as follows:
Input Acquisition: The user provides input via a web interface, which supports both text entry
and voice recording (using the Web Speech API for Speech-to-Text conversion).
Intent Processing (LLM Backend): The raw text is sent to a backend server running a Large
Language Model (LLM) via Ollama. This model is prompted to classify the user's intent into
two categories: "Informational Query" or "Vehicle Command."
Informational Queries (e.g., "What is vanadium e-fuel?") are answered directly by the LLM
using a retrieval-augmented generation (RAG) approach with project-specific context.

Figure 21 Logic for detecting the command.

Vehicle Commands (e.g., "Vandi, drive forward") are parsed into structured JSON instruction
sets and will be triggered only when the phase “Vandi” is detected.

Command Execution: Validated safe command instructions are forwarded to the Jetson Nano

4.6.2 Implementation
The implementation leverages a containerized microservices architecture to ensure portability
and ease of deployment.

24

Frontend: A responsive web-based chat interface was built using HTML/JavaScript, hosted
locally. It serves as the control panel for the demonstration.
Backend Processing: The core logic is managed by a python-based flask server that operates
the data flow between the speech services and the LLM (Ollama). This visual programming
approach allows for easy adjustment of the conversation logic without rewriting the
underlying codebase.
Deployment: The entire interactive stack runs inside a Docker container on an external PC
connected on the same Wi-Fi network as the Jetson Nano. This off-board processing strategy
ensures that the computationally intensive language tasks do not interfere with the critical
real-time navigation tasks running on the vehicle's onboard Jetson Nano or introduce more
power draw.

5. FUTURE WORK
While great progress has been made in validating the individual subsystems (fuel cell
chemistry, vehicle mechanics, and AI vision), in the next semester, we will focus on the final
manufacturing, integration, and full-system validation.

5.1 Cell Manufacture
The priority is to transition from the current PLA prototypes to a chemically robust final
stack .
Material upgrade: The final fuel cell stack will be manufactured using Polypropylene (PP) to
ensure long-term resistance to the highly acidic vanadium electrolyte, addressing the cracking
issues observed with PLA. Me and our BME teammate have already contacted one of the
manufacturers for the fuel cell tank in China, and we will conduct test on the manufactured
parts once it arrives.
Current collector fabrication process: I am still in the process of designing the current
collector and will contact manufacturer that can produce gold plated copper current collector
once I finish designing.
Final assembly: The stack will be assembled with the optimized Nafion N212 membrane and
sealed with PTFE gaskets to prevent cross-contamination and external leakage.

28

5.2 Final Assembly (cell and car)
This phase involves the physical integration of the power plant into the vehicle chassis.
Component Layout: The fuel cell stack will be mounted on to tray connected to the main car
body the tray will be designed once the cell stack design is settled.
Power Distribution: The electrical system will be wired to manage the dual power sources. A
lithium back up battery will be connected in parallel to ensure if the fuel cell is out of power,
the board would not immediately shut down and crash the system. So, we will implement a
power management system to ensure stable power input.
Refilling: The fuel refill will be finalized with chemically resistant tubing.

5.3 Auto Driving Program Implementation
The autonomous capabilities will be refined from basic detection to full closed-loop
navigation.
ROS Navigation Stack: The current object detection logic will be integrated into a complete
ROS navigation stack. The Jetson Nano will process LiDAR data for SLAM (Simultaneous
Localization and Mapping) and fuse it with the camera's visual data to make real-time path
planning decisions, we will adjust this plan based on the performance of the camera and
LiDAR.
Obstacle Avoidance: The control logic will be updated to actively react to traffic signals we
put on our track. There will be walls around the road that will be detected by the LiDAR, so
rather than just basic visual reactions, we want to enable dynamic navigation on the demo
track, but this will be further evaluated based on the performance and progress.

5.4 Interactive Interface Implementation
The chatbot frontend will be fully connected to the vehicle's control backend.
The voice-to-text, LLM processing, and command execution will be tested for latency and
accuracy to ensure the car responds to voice commands like "Stop," "Go to parking".
The LLM's context retrieval system will be updated with the latest project data, so it answers
all the technical questions about the project during the demo.

5.7 Acceptance Criteria
The project will be considered successful if the integrated “Vandi” prototype meets the
following measurable criteria during final integration and full-track testing.
Power system (vanadium fuel cell + power management)
Deliver ≥5 W continuous electrical output for ≥10 minutes while the car is driving on the
demo track (aligned with the stated ~5–7 W peak load requirement).
Provide a stable regulated 5 V rail to the Jetson/control electronics during motion, with no
unexpected resets when switching between fuel cell supply and the parallel lithium backup.
Pass a leak test with the final electrolyte (3 M sulfuric acid baseline) showing no visible
leakage at gaskets/tubing and no structural damage to the final chemically resistant stack
materials.
Mobility and manual fail-safe
Demonstrate full basic mecanum motions (forward/reverse/strafe/rotate/stop) under
remote/manual control, with emergency stop causing immediate halt (PWM → 0) reliably.
Operate with the fuel-cell tray installed without electrolyte spillage under normal
acceleration/braking used in the demo.
Perception (vision + traffic light state)
Achieve ≥80% correct detection for required demo objects (traffic lights + road signs) in the
demo environment, consistent with the stated target.
Run real-time perception on the onboard platform at ≥15 FPS end-to-end (YOLO + postprocessing), ensuring sufficient responsiveness for low-speed navigation.
Correctly classify traffic light state (red/green) using the YOLO + HSV pipeline in ≥90% of
trials at typical demo distances/lighting.
Autonomous navigation (mission-level)
Complete a full mission profile on the demo track: start from command → navigate through
traffic lights/obstacles → stop/park at designated zone, with ≥80% success rate over at least 5
consecutive trials.
Maintain safe obstacle behavior: if a wall/obstacle is detected (LiDAR/camera), the vehicle
slows/stops without contact in ≥90% of encounters during testing runs.
Interactive interface (chat/voice control)
Respond to at least 5 predefined voice/text commands (e.g., “Stop”, “Go to parking”) with
end-to-end latency ≤5 seconds from user input to vehicle action during live demo conditions
(same Wi‑Fi network).
Block “unsafe” commands consistently (as described in the safety-check logic) and provide a
clear rejection response instead of executing motion.
